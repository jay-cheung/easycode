# TUI Spec

## Objective

The TUI is an alternate terminal presentation for the existing CLI workflow. It must not fork agent execution, session persistence, permissions, slash command handling, plan approval, or logging.

## Required Coverage

- Session startup: default session creation, saved-session selection, explicit `--session`, and saved session updates.
- Slash commands: `/help`, `/settings`, `/sessions`, `/model`, `/thinking`, `/effort`, `/image`, `/skill`, and unknown/error output.
- Run control: normal prompts, queued input during active runs, `/cancel`, EOF, and `:exit`.
- Safety prompts: tool permission prompts, sandbox-bypass prompts, cancel-from-permission behavior, and remembered approvals.
- Plan workflow: proposed-plan persistence, approval, rejection, edit, new prompt, and transition from `plan` to `build`.
- Output: run start, reasoning, answer, tool calls/results, context compaction, repo-map prewarm, provider progress, provider metrics, and failures.
- Logger compatibility: `--logger --tui` still writes structured logs and transcript files while using the same run events.

## Current Contract

- `--tui` is explicit and opt-in.
- The TUI uses the same runner and data model as the plain CLI.
- The TUI shell shows mode, provider, model, session, logger state, root, current status, and command shortcuts.
- The TUI remains line-oriented so tests can exercise it without a pseudo-terminal or extra runtime dependency.

## Verification

- Unit tests cover TUI renderer status/panel/prompt behavior.
- CLI tests cover `--tui` argument parsing, interactive session commands, permission cancellation, and `--once` timeline output.
